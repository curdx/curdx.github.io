# product-manager

需求阶段负责人。把目标加研究产物翻译成结构化、可测试的需求。

## 它做什么

`product-manager` 在 `research.md` 通过审批后、运行 `/curdx-flow:requirements` 时被调用。它产出 `requirements.md`——含用户故事、功能需求、非功能需求、验收标准的结构化文档。

它**不会**凭空发明产品功能。它的工作是把研究浮现的事实结构化，让 `design.md` 有靠得住的输入。

## 触发时机

| 触发 | 行为 |
| --- | --- |
| `/curdx-flow:requirements` | 由目标 + `research.md` 生成 `requirements.md` |

## 产物：`requirements.md`

典型 `requirements.md` 含四个章节，每条带稳定 ID：

### 用户故事（`US-1`、`US-2` …）

```markdown
**US-1**：作为已认证用户，我可以用 OAuth 登录，避免管理密码。
```

用户故事让每条需求都锚定到一个人和一个目标。如果某条需求追溯不到任何用户故事，它大概率不该存在。

### 功能需求（`FR-1`、`FR-2` …）

```markdown
**FR-3**：系统必须支持有效期至少 30 天的 refresh token。
**FR-4**：每次刷新必须强制 token 轮换。
```

功能需求描述行为。它们是可测试的。

### 非功能需求（`NFR-1`、`NFR-2` …）

```markdown
**NFR-1**：预期负载下认证 P99 延迟必须小于 200 ms。
**NFR-2**：refresh token 静态存储必须加密。
```

NFR 覆盖性能、安全、持久性、可达性等横切属性。

### 验收标准（`AC-1`、`AC-2` …）

```markdown
**AC-1**：给定有效 refresh token，调用 /auth/refresh 时，发放新的 access token，并且 refresh token 被轮换。
```

验收标准是"完成"的契约。下游 `task-planner` 会把它们转成 `[VERIFY]` 门禁。

## 为什么稳定 ID 重要

每条需求都有稳定 ID（`US-3`、`FR-7`、`AC-4`）。下游产物会引用它们：

- `design.md` 标注每个组件满足哪些需求
- `tasks.md` 标注每个 `[VERIFY]` 任务测试哪些验收标准
- 代码评审时任何改动都能溯源回需求

需求被丢弃或重新编号，链就断了。`tasks.md` 已生成后，唯一安全的修订方式是 `refactor-specialist`。

## 实战示例

### 用户清晰的目标

```text
> 给被举报评论加一个管理员审核面板。
```

product-manager 会识别至少两类用户角色（管理员、审核员），每类多条用户故事，覆盖审核动作的功能需求，以及围绕审计日志的 NFR。

### 内部基础设施目标

```text
> 用新事件总线替换旧任务队列。
```

用户故事可以是面向开发者的（"作为服务作者，我可以发布事件而无需了解传输方式"）。内部用户也要写故事。

### 重构目标

```text
> 重构缓存层以提升可测性。
```

大部分需求是 NFR（可测性、隔离）和锁定当前行为的 AC，让重构可被证明等价。

## 阅读产物

审 `requirements.md` 时关注：

- **每条 US 应映射到真实的人**。"系统"这种模糊用户是味道。
- **每条 FR 应可测试**。"应当性能良好"不是需求；"1000 RPS 下 P99 < 200ms"是。
- **NFR 应覆盖研究浮现的横切关注**。研究标记的安全风险，如果没有 NFR 回应，要 push back。
- **验收标准要明确**。"功能正确"不是 AC。

## 最佳实践

- 通过审批前读完每条 AC。每一条都会成为自治循环要跑的 `[VERIFY]` 门禁。
- 早 push back。这一阶段补一条遗漏需求是一行编辑，到了 `implement` 之后就是几天返工。
- ID 保持稳定。重新编号会断掉下游溯源。
- 重构先把当前行为写成 AC，重构必须先满足这些 AC，才允许加新行为。
