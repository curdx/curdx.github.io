# research-analyst

研究阶段负责人。通过网络搜索、文档查询和代码库扫描调研目标——并以**并行团队**的方式运行，不是单 Agent。

## 它做什么

`research-analyst` 是 `/curdx-flow:start` 或 `/curdx-flow:research` 时调用的第一位专家。任务是把欠规约的目标转成有研究背景的简报，让后续阶段有可靠的输入。

具体地，研究阶段会：

1. 把目标拆成 2–4 个调研角度（每个角度一位组员）。
2. 派遣相同数量的 `research-analyst` 实例并行跑。
3. 每个实例针对自己的角度查文档、查网、查代码库。
4. 协调器把结果合并到 `research.md`。

## 触发时机

| 触发 | 行为 |
| --- | --- |
| `/curdx-flow:start` | 访谈后运行一次，再暂停审批 |
| `/curdx-flow:research` | 重新跑当前规约的研究阶段 |

## 产物：`research.md`

典型 `research.md` 包含：

- **执行摘要**（1–3 句，资深评审者 10 秒能读懂）
- **发现** 按调研角度分组
- **引用** 含来源 URL 和文档锚点
- **风险** 调研中识别出的，每条带简短缓解思路
- **可行性 / 风险 / 工作量** 整体方向评级
- **建议** 给下一阶段的指引

## 为什么并行

单一研究 Agent 串行调研。并行团队的好处是：

- 多个独立未知点的目标完成更快
- 每位组员只盯一个角度，避免上下文膨胀
- 早期暴露冲突——两位调研者意见相左时，合并步骤会显式记录分歧

## 实战示例

### 多个未知点的目标

```text
> 加一个带 token 刷新的 OAuth 登录。
```

可能的组员：

1. OAuth 流程细节（提供商文档）
2. Token 刷新策略和安全权衡
3. 代码库现有认证模式扫描

### 主要是代码库工作的目标

```text
> 重构缓存助手用新的 TTL 配置。
```

可能的组员：

1. 缓存助手当前状态 + 调用方
2. 新 TTL 配置 schema 与现有用法

### 主要是外部研究的目标

```text
> 支持 Anthropic 新的消息批量 API。
```

可能的组员：

1. Anthropic 批量 API 文档与限制
2. 同类 SDK 的批量模式

## 阅读产物

`research.md` 是规约最便宜的纠偏点。重点关注：

- **与目标矛盾的风险**。如果研究揭示某个约束让目标不可行，现在改方向——别等到 `tasks.md` 才回头。
- **模糊引用**。具体 URL 和章节锚点是好的，"各种博客"不是。
- **缺失的角度**。如果团队没调你认为关键的方向，重跑 `/curdx-flow:research` 时把它点名。

## 最佳实践

- 目标要具体。"加 OAuth"是一个角度，"加按租户限定的、带 refresh token 的 OAuth"是三个。
- 先读执行摘要。如果与你要的不符，研究就跑偏了，后续规约会继承漂移。
- 把风险列表当 TODO。研究里每条未处理的风险，到 `design.md` 通过时都应被回应。
