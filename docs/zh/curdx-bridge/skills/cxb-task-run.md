# cxb-task-run

通过 AutoFlow 流水线执行任务步骤。

## 概述

`cxb-task-run` 是 CurdX Bridge 的执行引擎。它逐步执行任务步骤，将文件操作委托给执行者（通常是 Codex），每步完成后运行审查，自动推进直到任务完成。

## 10 步流水线

### 步骤 1：同步状态

读取 `.curdx/state.json` 确定当前步骤。校验尝试次数限制并检查停止条件：

- 没有方案 → 提示你创建一个
- 所有步骤已完成 → 显示完成摘要

### 步骤 2：步骤设计（双独立设计）

Claude 和 Codex 分别独立设计当前步骤的方案：

- Claude 生成本地设计
- Codex 通过 `/cxb-ask` 生成独立设计
- Claude 合并二者：取完成条件、风险和方案的并集

合并后的设计包括：
- 方案描述
- 完成条件（最多 2 个，可验证）
- 风险和缓解措施
- 是否需要拆分为子步骤

### 步骤 3：拆分检查

如果步骤过大：
- 校验拟定的子步骤（3-7 个，原子化，有序）
- 在任务状态中应用拆分
- 从步骤 1 重新开始执行第一个子步骤

### 步骤 4-5：构建并发送 FileOpsREQ

构造结构化的文件操作请求，通过 `/cxb-file-op` 发送给执行者。Claude 不直接编辑文件 — 所有修改都通过执行者完成。

### 步骤 6：执行

执行者完成文件操作。根据配置路由到不同模式：

| 执行者配置 | 行为 |
|-----------|------|
| `codex` | Codex 直接执行 |
| `opencode` | Codex 监督 OpenCode 执行 |
| `codex+opencode` | Codex 负责读操作，OpenCode 负责写操作 |

### 步骤 7：处理响应

| 响应状态 | 动作 |
|---------|------|
| `ok` | 进入审查 |
| `ask` | 向你展示问题 |
| `fail` | 标记步骤为阻塞，附原因 |

### 步骤 8：审查

调用 [`cxb-review`](/zh/curdx-bridge/skills/cxb-review) 的 step 模式进行双重评估（Claude + 交叉审查）。

### 步骤 8.5：测试（可选）

Claude 根据变更类型决定是否需要测试。如需测试：
- 通过执行者运行测试
- 分析失败：修复 / 已知问题 / 阻塞

### 步骤 9：收尾

- 标记步骤完成
- 推进到下一步
- 更新 `todo.md` 和 `plan_log.md`
- 触发自动循环进入下一步

### 步骤 10：最终审查（仅任务完成时）

所有步骤完成后，运行 [`cxb-review`](/zh/curdx-bridge/skills/cxb-review) 的 task 模式：

- **小问题** — Codex 直接修复
- **中等问题** — 追加 1-2 个步骤
- **大问题** — 创建后续任务

生成包含执行摘要的最终报告。

## 设计原则

- **最短路径** — 最少的步骤和迭代
- **二值审查** — PASS 或 FIX，没有模糊地带
- **有限迭代** — 每步最多 2 次尝试，然后升级
- **自动推进** — 成功后自动进入下一步
