# 工作原理

## 架构概览

CurdX Bridge 使用 tmux（或 WezTerm）创建分屏终端布局。每个 AI Provider 运行在自己的面板中，轻量级守护进程管理它们之间的通信。

![CurdX Bridge 布局](/images/curdx-bridge/layout.svg)

```
┌──────────────────────┬──────────────┐
│                      │    Codex     │
│       Claude         ├──────────────┤
│     （主面板）        │   Gemini     │
│                      ├──────────────┤
│                      │  OpenCode    │
└──────────────────────┴──────────────┘
```

- **左侧面板** — Claude，你的主要界面。在这里输入。
- **右侧面板** — Codex、Gemini、OpenCode。可以实时观察它们的工作过程。

## 通信模型

当你说"让 Codex 审查下"，背后发生了什么：

1. Claude 调用 `/cxb-ask` 技能发送你的请求
2. 请求通过异步协议发到 Codex 的面板
3. Codex 处理请求 — 你可以在它的面板中看到它在工作
4. Claude 通过 `/cxb-reply` 获取响应
5. 结果出现在你和 Claude 的对话中

每个 Provider 独立运行。它们不直接共享上下文 — Claude 作为协调者，决定发送什么信息以及如何整合响应。

## 角色体系

CurdX Bridge 为每个 AI 分配一个角色，定义其职责：

| 角色 | 默认 Provider | 职责 |
|------|--------------|------|
| **设计师** | Claude | 规划架构、编写代码、协调其他 Agent |
| **评审员** | Codex | 使用评分 Rubric（每维度 1-10）评估代码和方案 |
| **灵感源** | Gemini | 生成创意替代方案和头脑风暴思路（仅供参考） |
| **协作者** | OpenCode | 按需提供额外 AI 视角 |

### 角色分配

角色在项目的 `CLAUDE.md` 文件中配置：

```markdown
| Role | Provider | Description |
|------|----------|-------------|
| designer | claude | 主要规划师和架构师 |
| inspiration | gemini | 创意头脑风暴 |
| reviewer | codex | 评分质量门禁 |
| executor | claude | 代码实现 |
```

可以通过编辑 Provider 列来重新分配角色。例如，让 Gemini 代替 Codex 做评审。

## 审查框架

审查系统使用两套 Rubric 作为质量门禁：

### Rubric A — 方案审查

5 个维度，每项 1-10 分：

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| 清晰度 | 20% | 其他开发者能否无疑问地按步骤执行？ |
| 完整性 | 25% | 是否覆盖所有需求、边界情况和交付物？ |
| 可行性 | 25% | 步骤是否在现有代码库中可实现？ |
| 风险评估 | 15% | 风险是否已识别并有具体缓解措施？ |
| 需求对齐 | 15% | 每个步骤是否都能追溯到明确的需求？ |

### Rubric B — 代码审查

6 个维度，每项 1-10 分：

| 维度 | 权重 | 评估内容 |
|------|------|---------|
| 正确性 | 25% | 代码是否实现了方案要求的功能？ |
| 安全性 | 15% | 是否存在注入、硬编码密钥或缺失校验？ |
| 可维护性 | 20% | 代码整洁、命名规范、遵循项目约定？ |
| 性能 | 10% | 是否存在不必要的复杂度或阻塞调用？ |
| 测试覆盖 | 15% | 变更路径是否有通过的测试覆盖？ |
| 方案一致性 | 15% | 实现是否与批准的方案一致？ |

### 通过/失败标准

- **通过**：加权总分 ≥ 7.0 且无任何单项 ≤ 3
- **失败**：修复标记的问题后重新提交（最多 3 轮）
- **升级**：3 轮失败后，结果呈现给你做最终决定

## 异步协议

Provider 通信使用异步请求/响应模式：

1. **请求** — Claude 通过 `cxb-ask` 向 Provider 面板发送消息
2. **处理** — Provider 在自己的面板中工作（你可以看到）
3. **响应** — Claude 通过 `cxb-reply` 获取结果

这意味着：
- Provider 之间互不阻塞
- 你可以实时观察每个 Provider 的工作
- Claude 在等待响应时保持可用
- 每个 Provider 保持独立的会话上下文

## 会话管理

每个 CurdX 会话为选定的 Provider 创建 tmux 面板。会话可以：

- **创建** — `curdx [providers...]` 启动新会话
- **恢复** — `curdx -r` 恢复之前的会话，上下文完整保留
- **终止** — `curdx kill` 终止所有面板，`curdx kill <provider> -f` 强制终止指定面板

会话状态存储在项目目录的 `.curdx/` 中。
